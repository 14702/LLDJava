Journey State Machine — Allowed Transitions
CREATED
ACTIVE
PAUSED
← PAUSED can go back to ACTIVE →
EXPIRED
COMPLETED
ACTIVE → EXPIRED (auto, when time-bound end-date passes)
ACTIVE → COMPLETED  |  PAUSED → COMPLETED
EXPIRED and COMPLETED are terminal — no further transitions allowed
Transition table enforced via ALLOWED_TRANSITIONS map in JourneyServiceImpl.
Invalid transitions throw JourneyException(INVALID_STATE_TRANSITION).
evaluate(userId, payload) — Decision Flow
1. Iterate all journeys
For each journey in repository, auto-expire if needed. Skip if state ≠ ACTIVE or if startDate hasn't passed yet.
↓
2. Lookup UserJourneyRecord
Find the latest active record for (userId, journeyId). This tells us if user is already onboarded.
↓
Branch based on record state:
A. No record → Try Onboard
Match payload against journey's onboardingCondition.
If match: create UserJourneyRecord at onboarding stage, notify observers.
B. At terminal + recurring → Re-onboard
User completed the journey but it's recurring.
Try onboarding again with a new runNumber.
C. Has record, not terminal → Try Transition
Scan transitions where fromStageId == currentStageId.
If condition matches payload: move user to toStageId, notify observers.
↓
3. No match → Ignored
If payload doesn't match any onboarding condition or transition condition, nothing happens. Silent no-op.
